## Summary

### Task 7: Memory 시스템 구현

Open Design의 Memory 개념을 neos-work에 이식. 에이전트 실행 간 key-value 형태의 기억을 영속 저장.

#### 서버 (DB + API)

- **`apps/server/src/db/schema.ts`**
  - `memory` 테이블 생성:
    ```sql
    CREATE TABLE IF NOT EXISTS memory (
      id TEXT PRIMARY KEY,
      key TEXT NOT NULL UNIQUE,
      value TEXT NOT NULL,
      created_at TEXT NOT NULL,
      updated_at TEXT NOT NULL
    )
    ```

- **`apps/server/src/db/memory.ts`** (신규)
  - `listMemory()` — `SELECT * FROM memory ORDER BY updated_at DESC`
  - `upsertMemory(key, value)` — INSERT OR REPLACE with `updated_at = NOW()`
  - `deleteMemory(id)` — `DELETE FROM memory WHERE id = ?`

- **`apps/server/src/routes/memory.ts`** (신규)
  - `GET /` — 전체 메모리 목록 반환
  - `PUT /:key` — key-value upsert
  - `DELETE /:id` — 단건 삭제

- **`apps/server/src/index.ts`** — `/api/memory` 라우트 등록

#### 클라이언트

- **`apps/desktop/src/lib/engine.ts`**
  - `MemoryItem` 인터페이스: `{ id, key, value, createdAt, updatedAt }`
  - `listMemory()`, `upsertMemory(key, value)`, `deleteMemory(id)` 메서드 추가

- **`apps/desktop/src/pages/Memory.tsx`** (신규)
  - 전체 목록 표시 (key · value · 수정일)
  - 인라인 "Add Memory" 입력 폼 (key + value 텍스트 필드)

## Architecture Notes

### OpenAI 어댑터 설계 결정

OpenAI API와 Ollama는 동일한 `/v1/chat/completions` 스펙을 공유하므로 단일 `OpenAIAdapter` 클래스로 두 프로바이더를 처리한다. `provider` 생성자 인자로 분기:
- `'openai'`: apiKey 필수, `Authorization: Bearer` 헤더 포함
- `'ollama'`: apiKey 불필요, 로컬 서버 endpoint 사용

### Memory 시스템 설계 결정

- 현 단계에서는 전역 메모리(workspace-scoped key-value store)만 구현. 세션/워크플로우 단위 scoping은 v0.3.0에서 AgentNode 통합 시 추가 예정.
- 에이전트에서 Memory를 자동으로 읽고 쓰는 Tool은 v0.3.0 대상 (`memory_read`, `memory_write` tool 블록).

### SKILL.md 파싱 호환성

기존 SKILL.md 파일(name/description만 있는 경우)은 신규 필드를 `undefined`로 처리하므로 하위 호환성 유지. Open Design ecosystem의 확장 frontmatter를 가진 파일은 모든 필드가 DB에 저장되고 UI에 표시된다.
