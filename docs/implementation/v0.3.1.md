# v0.3.1 Implementation

## Summary

v0.3.0 Open Design 마이그레이션 계획(PLAN_FOR_V0_3_0) 중 남아 있던 **Deploy 노드 이력·설정 UI** 공백을 해소한다.
배포 이력 테이블/API, Settings Deploy 토큰 섹션, Deployments 페이지, 워크플로우 연동 메타데이터 저장을 포함한다.

기준: v0.3.0 (Design Context, Routine, CLI spawn, Artifact Preview, Plugin, Media/Deploy 노드, MCP OAuth, ZIP, 병렬 게이트, 코딩 블록, Webhook, 레이아웃, revision 완료)

---

## Changes

### Task 1: Deployments DB + API history (PLAN Task 8 remainder)

**개념**: Deploy 노드 실행 결과를 SQLite에 기록하고 목록/삭제 API를 제공한다.

- **`apps/server/src/db/deployments.ts`** (신규)
  - `deployments` 테이블 매핑: `id`, `workflow_id`, `run_id`, `provider`, `project_name`, `url`, `deployment_id`, `status`, `status_message`, timestamps
  - `createDeployment()`, `getDeployment()`, `listDeployments({ workflowId?, limit? })`, `updateDeployment()`, `deleteDeployment()`

- **`apps/server/src/db/schema.ts`** (수정)
  - `CREATE TABLE IF NOT EXISTS deployments` + 인덱스 (`workflow_id`, `created_at`)

- **`apps/server/src/routes/deploy.ts`** (수정)
  ```
  GET    /api/deploy              # 전체 이력 (?workflowId=&limit=)
  GET    /api/deploy/:id
  DELETE /api/deploy/:id
  POST   /api/deploy              # 기존 배포 + 이력 기록
  ```
  - POST 시 `status: deploying` 레코드 생성 후 성공/실패에 따라 `updateDeployment`
  - body에 optional `workflowId`, `runId` 수용

## Architecture Notes

- Deploy 실행 경로: DeployNode → `POST /api/deploy` → Vercel/Cloudflare API → `deployments` 행 갱신
- 목록 UI는 서버 토큰 없이 이력만 조회; 실제 배포 자격 증명은 Settings에만 보관
- 기존 `POST /api/deploy` 계약은 유지하고 `recordId`를 data에 추가
