# v0.2.0 구현 문서 — 워크플로우 빌더

**릴리즈:** 2026-05-10  
**기준 계획:** `docs/plans/PLAN_FOR_V0_2_0.md`  
**기반 버전:** v0.1.4 (Audit 완료)

## 구현 요약

v0.2.0에서는 n8n 스타일의 비주얼 워크플로우 빌더, 에이전트 하네스 시스템, 도메인 블록(Domain Block) 시스템을 구현했다. 기존 단일 세션/단일 에이전트 구조에 **워크플로우 레이어**가 추가되었으며, React Flow 기반 에디터, 새로운 `workflow-engine` 패키지, 서버 라우트, SQLite 테이블 확장이 포함된다.

## 2. 핵심 인터페이스

### 2.1 노드 실행 계약

```typescript
// packages/workflow-engine/src/types.ts
export interface NodeContext {
  workflowId: string;
  runId: string;
  nodeId: string;
  inputs: Record<string, unknown>;   // 상위 노드 출력 결합
  settings: Record<string, string>;  // API 키 등 (settings DB에서 주입)
  config?: Record<string, unknown>;  // 노드별 설정값
  signal?: AbortSignal;
}

export interface NodeResult {
  ok: boolean;
  output: unknown;
  error?: string;
  durationMs: number;
}

export interface ExecutableNode {
  type: NodeType;
  execute(ctx: NodeContext): Promise<NodeResult>;
}
```

- `WorkflowNode`(그래프 데이터 타입, `@neos-work/shared`)와의 이름 충돌을 피하기 위해 실행 가능 노드 인터페이스는 `ExecutableNode`로 명명했다.
- LLM 어댑터는 `AgentNode.execute()` 내부에서 `settings`를 읽어 직접 생성한다. `executor.ts`가 어댑터를 미리 생성해 컨텍스트에 주입하지 않는다.

## 3. DAG 실행 엔진 (`executor.ts`)

### 3.1 위상정렬

`graph.ts`의 `topologicalSort`가 Kahn's BFS 알고리즘으로 노드 순서를 결정한다. 사이클이 존재하면 `'Workflow contains a cycle'` 에러를 던져 실행을 중단한다.

### 3.2 실행 흐름

1. `topologicalSort`로 노드 순서 결정
2. 위상정렬 순서대로 노드를 **순차 실행**
3. 각 노드의 `output`을 `nodeOutputs` Map에 저장
4. 다음 노드의 `inputs`는 인입 엣지(incoming edges)의 소스 노드 출력을 결합

### 3.3 실패 전파 (`failedNodes`)

```typescript
const failedNodes = new Set<string>();
```

- 노드가 `ok: false`를 반환하면 해당 노드 ID를 `failedNodes`에 추가
- **AND 게이트**: 업스트림 중 하나라도 실패 → 게이트 실패
- **OR 게이트**: 업스트림 전체가 실패한 경우에만 게이트 실패; 하나라도 성공이면 첫 번째 성공 입력 사용
- **일반 노드**: 모든 직접 업스트림이 실패한 경우 `'Skipped: all upstream nodes failed'`로 실패 처리

> 계획 문서 대비 추가된 동작이다. 계획에는 실패 전파 정책이 명시되지 않았으나, 실패한 노드의 출력을 하위 노드에 전달하면 잘못된 데이터로 실행될 수 있어 명시적 skip 정책을 추가했다.

### 3.4 출력 크기 제한

```typescript
const MAX_OUTPUT_BYTES = 1_048_576; // 1 MB
```

직렬화된 노드 출력이 1 MB를 초과하면 `{ truncated: true, preview: ... }` 로 잘라낸다.

### 3.5 SSE 이벤트 타입

| 이벤트 | 발생 시점 |
|--------|-----------|
| `run.started` | 실행 시작 |
| `node.started` | 개별 노드 실행 시작 |
| `node.completed` | 노드 성공 완료 |
| `node.failed` | 노드 실패 또는 skip |
| `run.completed` | 전체 실행 완료 |
| `run.failed` | 실행 중 예외 발생 |

## 4. 노드 구현

### 4.1 AgentNode (`nodes/agent.ts`)

- `settings['llmProvider']` 값에 따라 `AnthropicAdapter` 또는 `GoogleAdapter`를 선택
- `harnessId`가 있으면 `resolveHarness`로 하네스를 조회하고 `systemPrompt` + `allowedTools` + `maxSteps`를 주입
- 하네스 systemPrompt와 노드 레벨 systemPrompt는 `\n\n---\n`으로 결합 (하네스가 기반, 노드가 추가 문맥)

### 4.2 BlockNode (`nodes/block.ts`)

세 가지 구현 방식을 분기 처리한다:

| implementationType | 처리 방식 |
|-------------------|-----------|
| `native` | `NativeBlockExecutor.execute()` 직접 호출 |
| `prompt` | 템플릿에 `{{params}}` / `{{inputs}}` 치환 후 AgentNode에 위임 |
| `skill` | `Use skill: {skillId}` 프롬프트로 AgentNode에 위임 |

프롬프트/스킬 방식에서 `ctx.inputs`를 직렬화해 삽입하기 전에 `MAX_INPUT_LENGTH = 4096` 바이트로 잘라 프롬프트 인젝션을 완화한다.

### 4.3 Gate 노드 (`nodes/gate.ts`)

- **TriggerNode**: `ctx.inputs`를 그대로 `output`으로 반환
- **OutputNode**: 모든 상위 입력을 `Object.assign`으로 병합해 반환
- **AndGateNode**: 모든 상위 입력을 병합 (실패 전파는 executor가 처리)
- **OrGateNode**: 첫 번째 입력만 선택해 반환

> **v0.2.0 실행 모델 주의**: 현재는 순차 실행이므로 OR 게이트의 "첫 번째 완료된 입력 선택" 동작은 단순히 위상정렬 순서상 첫 번째 업스트림 결과를 선택한다. 병렬 실행(Promise.all)은 v0.3.0 예정.

### 4.4 WebSearchNode (`nodes/web-search.ts`)

- `ctx.inputs['query']` 또는 `ctx.inputs['text']`를 검색 쿼리로 사용
- `TAVILY_API_KEY` 필요; 미설정 시 즉시 실패
- 결과: `{ title, url, content, score }[]` (최대 5건)

### 4.5 SlackMessageNode (`nodes/slack.ts`)

- `@slack/web-api` SDK 사용
- `SLACK_BOT_TOKEN` 필요; `ctx.config['channel']` 또는 `ctx.inputs['channel']`에서 채널 읽기
- `ctx.inputs['text']` 없으면 전체 inputs를 JSON 직렬화해 전송

### 4.6 DiscordMessageNode (`nodes/discord.ts`)

- 라이브러리 없이 HTTP Webhook 직접 호출
- **SSRF 방어**: `DISCORD_WEBHOOK_URL`이 `https://discord.com/api/webhooks/` 로 시작하지 않으면 즉시 `Invalid Discord webhook URL` 오류 반환

## 5. 블록 시스템

### 5.1 블록 레지스트리 (`blocks/registry.ts`)

```typescript
const builtInRegistry = new Map<string, NativeBlockExecutor>(); // 실행자
const metaRegistry = new Map<string, WorkflowBlock>();          // 메타데이터
```

- `registerNativeBlock(executor, meta?)`: 실행자와 메타를 각각 등록
- `resolveBlock(id)`: 메타 조회
- `getNativeExecutor(id)`: 실행자 조회
- `listBlocks(domain?)`: 도메인 필터 가능한 전체 목록 반환

### 5.2 금융 내장 블록 (`blocks/finance/index.ts`)

> **계획 대비 변경사항**: 계획 문서의 스크리닝 중심 블록(`finance.box_breakout`, `finance.volume_surge` 등) 대신, 지표 계산/데이터 조회 중심 블록으로 구현하였다. 개별 지표 블록을 체이닝해 스크리닝 파이프라인을 조립하는 방식으로 유연성을 높였다.

| 블록 ID | 이름 | 카테고리 | 필수 파라미터 | 출력 |
|---------|------|----------|--------------|------|
| `price_lookup` | Stock Price Lookup | `market-data` | `symbol` | `{ symbol, currentPrice, changePercent, volume, ... }` |
| `moving_average` | Moving Average | `technical-indicator` | `symbol`, `period?`, `type?` | `{ symbol, type, period, current, series }` |
| `rsi` | RSI | `technical-indicator` | `symbol`, `period?` | `{ symbol, period, current, signal, series }` |
| `macd` | MACD | `technical-indicator` | `symbol`, `fastPeriod?`, `slowPeriod?`, `signalPeriod?` | `{ symbol, current, histogram, signal, series }` |
| `portfolio_summary` | Portfolio Summary | `portfolio` | `symbols` (최대 20) | `{ count, totalValue, items, gainers, losers }` |
| `risk_report` | Risk Report | `risk` | `symbol`, `lookback?` | `{ dailyVolatility, annualisedVolatility, maxDrawdown, sharpeRatio, riskLevel }` |

모든 금융 블록은 `KIS_APP_KEY`, `KIS_APP_SECRET` 설정이 필요하다.

**기술 지표 라이브러리:** `technicalindicators` (SMA, EMA, RSI, MACD 계산)

### 5.3 KIS API 클라이언트 (`blocks/finance/kis-api.ts`)

- **OAuth2 토큰 발급**: `POST /oauth2/tokenP` → Access Token 발급
- **토큰 캐시**: 모듈 레벨 Map에 캐시, 만료 1분 전 자동 갱신
- **캐시 키 보안**: `appKey + appSecret`을 SHA-256 해시하여 Map 키로 사용 (평문 노출 방지)
- **날짜 계산**: `getStockChart` 호출 시 요청 봉 수에 ×1.5 배수를 적용해 주말·공휴일로 인한 데이터 부족을 방지

```typescript
// 수정된 날짜 계산 (주말 여유분 포함)
const calendarDays = Math.ceil(count * 1.5);
const startDate = new Date(today.getTime() - calendarDays * 24 * 60 * 60 * 1000)
  .toISOString().slice(0, 10).replace(/-/g, '');
```

### 6.3 하네스 주입 흐름

```
WorkflowNode.config.harnessId = 'finance_analyst'
    ↓
AgentNode.execute() → resolveHarness(harnessId)
    ↓
systemPrompt = harness.systemPrompt + "\n\n---\n" + node.config.systemPrompt
allowedTools = harness.allowedTools
maxIterations = harness.constraints.maxSteps
    ↓
AgentOrchestrator.run(goal, signal)
```

| `/api/blocks` | `routes/blocks.ts` | GET, POST |

### 7.2 워크플로우 실행 (SSE 스트림)

`POST /api/workflow/:id/run`은 Hono `stream()`으로 SSE 스트림을 열고 `executeWorkflow`를 구동한다.

- 실행 시작 시 `workflow_run` 테이블에 `status='running'` 레코드 생성
- 완료/실패 시 `status` 업데이트 및 `node_results_json` 저장
- 클라이언트 연결 끊김 시 `c.req.raw.signal.abort` → `AbortController`로 실행 취소

### 7.3 하네스 보호 정책

내장 하네스(`isBuiltIn: true`)는 PUT/DELETE 요청에 403을 반환한다.

```typescript
if (resolveHarness(id)?.isBuiltIn) {
  return c.json({ ok: false, error: 'Cannot modify built-in harness' }, 403);
}
```

### 7.4 커스텀 블록 native 스텁

`POST /api/blocks`로 `implementationType: 'native'` 커스텀 블록을 생성하면, 실제 실행자가 없으므로 서버 재시작 없이 실행할 수 없음을 알리는 스텁 실행자를 등록한다.

### 7.5 DB 스키마 변경 (`db/schema.ts`)

v0.2.0에서 추가된 테이블:

```sql
CREATE TABLE IF NOT EXISTS workflow (

CREATE TABLE IF NOT EXISTS workflow_run (

  completed_at      TEXT,
  error             TEXT

CREATE TABLE IF NOT EXISTS custom_harness ( ... );
CREATE TABLE IF NOT EXISTS custom_block ( ... );
```

- 상단 툴바: 저장(PUT 워크플로우) / 실행(SSE 스트림 시작) / 중지(AbortController)
- 우측 패널: SSE 이벤트 실시간 로그 표시
- 노드 색상 코딩으로 실행 상태 시각화 (실행 중: 노란색 테두리, 완료: 초록색, 실패: 빨간색)

**config 보존 설계:**

노드의 `config` 값이 React Flow 렌더링 사이클에서 유실되지 않도록 `data.config`에 저장하고, 저장 시 `n.data.config`를 읽어 서버에 전달한다.

```typescript
// 저장 시 config 복원
nodes: nodes.map((n) => ({
  ...
  config: (n.data.config as Record<string, unknown>) ?? {},
}))
```

### 8.3 타입 정의 (`lib/engine.ts`)

클라이언트 측 타입을 서버/공유 타입과 일치하도록 `WorkflowBlock.paramDefs` 구조를 수정했다.

```typescript
// 수정 전 (불일치)
paramDefs: Array<{ name: string; type: string; ... }>

// 수정 후 (공유 타입과 일치)
paramDefs: Array<{ key: string; label: string; type: string; ... }>
```
