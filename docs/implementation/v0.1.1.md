## Overview

v0.1.1 is an incremental release that delivers Tier 1 (performance fixes) and Tier 2 (agent core) items from [NEXT_STEPS.md](../NEXT_STEPS.md). It also retroactively documents the security hardening (S1, S2) that was applied as part of the post-v0.1.0 code review commit (`e00a2b7`) but not captured in v0.1.0 documentation.

### What v0.1.1 Adds

- **S1** API key encryption at rest (AES-256-GCM, machine-derived key)
- **S2** Bearer token authentication middleware on all API routes
- **S3** Message list virtualization (`react-virtuoso`)
- **S4** ReactMarkdown streaming performance — deferred parse + memoization
- **A3** `agent_step` DB table with CRUD helpers
- **A4** Server-side cancel API (`POST /api/session/:id/cancel`)
- **A1/A2** Agent orchestrator and task planner (`packages/core/src/agent/`)

### What v0.1.1 Does NOT Include

- MCP client (`packages/mcp-client/`)
- Browser automation (Playwright)
- Skill system (parser, discovery, UI)
- Server-side agent route integration (orchestrator is `packages/core` only)
- CI/CD pipeline

## Security Hardening (S1, S2)

These fixes were implemented in commit `e00a2b7` (post-v0.1.0) but not documented. They resolve the Critical issues listed in the v0.1.0 Known Issues section.

### S1: API Key Encryption at Rest

**File**: `apps/server/src/db/crypto.ts` (new), `apps/server/src/db/settings.ts` (modified)

API keys are now encrypted before storage using AES-256-GCM with a machine-derived key:

```
Key derivation: SHA256(hostname + homedir + "neos-work-v1")
Stored format:  <iv_hex>:<authTag_hex>:<ciphertext_hex>
Sensitive keys: any key matching /^apiKey\./
```

The `settings.ts` module transparently encrypts on write and decrypts on read:

```typescript
// Write path
setSetting('apiKey.anthropic', plaintext)
→ encrypt(plaintext)  // produces "iv:tag:cipher"
→ INSERT INTO setting ...

// Read path
getSetting('apiKey.anthropic')
→ SELECT value FROM setting ...
→ isEncrypted(value) ? decrypt(value) : value

// Migration (runs on server startup)
migrateEncryption()
→ find all apiKey.* rows without encrypted format
→ re-encrypt in-place
```

**Limitation**: This is machine-local encryption, not OS keychain integration. The derived key is deterministic from system properties, so it provides protection against direct DB file access but not against an attacker with full system access. Tauri keychain integration remains a future improvement.

### S2: Bearer Token Authentication

**File**: `apps/server/src/index.ts`

Two authentication layers are applied to all non-health-check requests:

**Layer 1 — Host header validation** (DNS rebinding protection):
```typescript
// Rejects requests whose Host header isn't a known local origin
const allowed = new Set(['localhost', '127.0.0.1', 'tauri://localhost']);
if (!allowed.has(host)) return c.json({ ok: false, error: 'Forbidden' }, 403);
```

**Layer 2 — Bearer token validation**:
```typescript
// A 32-byte random token is generated at server startup
const authToken = crypto.randomBytes(32).toString('hex');
// Output to stdout so Tauri sidecar can capture it
console.log(`NEOS_AUTH_TOKEN=${authToken}`);

// Middleware checks Authorization header on every request
const authHeader = c.req.header('Authorization');
if (authHeader !== `Bearer ${authToken}`) {
  return c.json({ ok: false, error: 'Unauthorized' }, 401);
}
```

The desktop app reads `NEOS_AUTH_TOKEN` from the sidecar stdout and stores it via `EngineClient.setAuthToken()`.

## Performance Improvements (S3, S4)

### S3: Message List Virtualization

**File**: `apps/desktop/src/pages/Sessions.tsx`
**Dependency**: `react-virtuoso@^4.18.3`

The `ChatArea` component previously rendered all messages to the DOM via `.map()`, causing linear performance degradation for long sessions.

**Before**:
```tsx
<div className="flex-1 overflow-auto p-6">
  {messages.map((msg) => (
    <div key={msg.id} …>…</div>
  ))}
  <div ref={messagesEndRef} />
</div>
```

**After**:
```tsx
<Virtuoso
  style={{ flex: 1 }}
  data={messages}
  followOutput="smooth"
  itemContent={(_, msg) => (
    <div className="px-6 mb-4 …">…</div>
  )}
/>
```

Key behavioral changes:
- `followOutput="smooth"`: auto-scrolls to new messages during streaming; suspends auto-scroll if user manually scrolls up
- `messagesEndRef` + `useEffect` scroll management removed (handled by Virtuoso internally)
- Only the messages visible in the viewport are kept in the DOM; off-screen messages are unmounted

**Why `react-virtuoso` over `react-window`**: Chat UIs require variable-height items and bottom-anchored scroll behavior. `react-window`'s `VariableSizeList` requires manually measuring and caching each item's height. `react-virtuoso` handles this automatically.

### S4: ReactMarkdown Streaming Performance

**File**: `apps/desktop/src/pages/Sessions.tsx`

`<ReactMarkdown>` parses the full markdown AST on every render. During streaming, this happens on every text chunk — O(n) parsing per character added.

**Strategy**: Deferred parse + memoization

```tsx
const MarkdownContent = memo(function MarkdownContent({
  content,
  isStreaming,
}: {
  content: string;
  isStreaming?: boolean;
}) {
  if (isStreaming) {
    // Raw text during streaming — no AST parsing
    return (
      <pre className="whitespace-pre-wrap text-sm leading-relaxed"
           style={{ fontFamily: 'inherit' }}>
        {content}
      </pre>
    );
  }
  // Full markdown parse only once, after streaming completes
  return (
    <div className="markdown-content">
      <ReactMarkdown remarkPlugins={…} rehypePlugins={…}>
        {content}
      </ReactMarkdown>
    </div>
  );
});
```

Two optimizations combined:
1. **Deferred parsing** — `<pre>` during streaming avoids repeated AST construction
2. **`React.memo`** — skips re-render entirely if `content` and `isStreaming` are reference-equal

The `isStreaming` prop is passed down from the `DisplayMessage.isStreaming` field that the chat stream handler already maintains.

## Agent Step DB Table (A3)

### Schema Addition

**File**: `apps/server/src/db/schema.ts`

```sql
CREATE TABLE IF NOT EXISTS agent_step (
  id          TEXT PRIMARY KEY,
  session_id  TEXT NOT NULL REFERENCES session(id) ON DELETE CASCADE,
  step_index  INTEGER NOT NULL,
  type        TEXT NOT NULL,   -- 'plan' | 'tool_use' | 'tool_result' | 'reasoning' | 'error'
  status      TEXT NOT NULL DEFAULT 'pending',
                               -- 'pending' | 'running' | 'completed' | 'error'
  data        TEXT,            -- JSON (step input/output)
  error       TEXT,
  created_at  TEXT DEFAULT (datetime('now')),
  updated_at  TEXT DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_agent_step_session_id ON agent_step(session_id);
CREATE INDEX IF NOT EXISTS idx_agent_step_status ON agent_step(status);
```

### CRUD Helpers

**File**: `apps/server/src/db/agent-steps.ts` (new)

| Function | Signature | Description |
|----------|-----------|-------------|
| `createAgentStep` | `(params) → AgentStepRow` | Insert a new pending step |
| `getAgentStep` | `(id) → AgentStepRow \| undefined` | Fetch single step by ID |

## Cancel API (A4)

### Server Side

**File**: `apps/server/src/routes/session.ts`

A module-level `Map<string, AbortController>` tracks active chat streams:

```typescript
const activeChats = new Map<string, AbortController>();
```

**Registration (on chat start)**:
```typescript
const serverAbort = new AbortController();
activeChats.set(sessionId, serverAbort);

// Merge with client-disconnect signal using AbortSignal.any()
const signals: AbortSignal[] = [serverAbort.signal];
if (c.req.raw.signal) signals.push(c.req.raw.signal);
const abortSignal = AbortSignal.any(signals);
```

**Cleanup (on chat end)**:
```typescript
} finally {
  activeChats.delete(sessionId);
}
```

**Cancel endpoint**:
```typescript
session.post('/:id/cancel', (c) => {
  const sessionId = c.req.param('id');
  const controller = activeChats.get(sessionId);
  if (!controller) return c.json({ ok: false, error: 'No active chat for session' }, 404);
  controller.abort();
  return c.json({ ok: true });
});
```

`AbortSignal.any()` ensures the combined signal fires on whichever happens first — client tab close, network disconnect, or explicit cancel request.

### Client Side

**File**: `apps/desktop/src/lib/engine.ts`

```typescript
async cancelSession(sessionId: string): Promise<ApiResponse<void>> {
  const res = await fetch(`${this.baseUrl}/api/session/${sessionId}/cancel`, {
    method: 'POST',
    headers: this.getHeaders(),
  });
  return res.json();
}
```

**Note**: The desktop UI (`Sessions.tsx`) still uses `abortRef.current?.abort()` to cancel the fetch SSE connection from the client side. The new `cancelSession()` method provides a complementary server-side signal for scenarios where the client connection remains open (e.g., network routing anomalies).

## Agent Package (A1/A2)

**Location**: `packages/core/src/agent/`

```
packages/core/src/agent/
├── types.ts        — AgentStep, AgentTask, AgentEvent, OrchestratorOptions
├── planner.ts      — Planner class (A2)
├── orchestrator.ts — AgentOrchestrator class (A1)
└── index.ts        — barrel export
```

Exported from `packages/core` via `packages/core/src/index.ts`.

### Types (`types.ts`)

```typescript
type AgentStepType   = 'plan' | 'tool_use' | 'tool_result' | 'reasoning' | 'error';
type AgentStepStatus = 'pending' | 'running' | 'completed' | 'error';
type AgentTaskStatus = 'idle' | 'running' | 'completed' | 'failed' | 'cancelled';

interface AgentStep {
  id: string;
  index: number;
  description: string;
  type: AgentStepType;
  status: AgentStepStatus;
  toolName?: string;
  input?: Record<string, unknown>;
  output?: unknown;
  error?: string;
}

interface AgentTask {
  id: string;
  goal: string;
  steps: AgentStep[];
  status: AgentTaskStatus;
  createdAt: Date;
  completedAt?: Date;
}

// Events yielded by AgentOrchestrator.run()
type AgentEvent =
  | { type: 'plan';          steps: AgentStep[]             }
  | { type: 'step_start';    step: AgentStep                }
  | { type: 'step_complete'; step: AgentStep                }
  | { type: 'step_error';    step: AgentStep; error: string }
  | { type: 'text';          content: string                }
  | { type: 'done';          task: AgentTask                }
  | { type: 'error';         error: string                  };
```

### Planner (`planner.ts`)

The `Planner` class decomposes a natural language goal into an ordered list of `AgentStep` objects using an LLM call:

```typescript
class Planner {
  constructor(adapter: LLMProviderAdapter) {}

  async plan(goal: string, context?: string, signal?: AbortSignal): Promise<AgentStep[]>
}
```

**System prompt** instructs the LLM to return a JSON array:
```json
[{ "description": "…", "toolName": "optional_tool" }]
```

Parsing is resilient: the method extracts the first `[…]` block from the response with a regex, falls back to a single "Execute the goal directly" step if JSON parsing fails.

**Model selection**: uses `adapter.getModels()[0]` — the first model returned by the injected adapter.

### Orchestrator (`orchestrator.ts`)

The `AgentOrchestrator` implements a ReAct (Reasoning + Acting) loop:

```typescript
class AgentOrchestrator {
  constructor(
    adapter: LLMProviderAdapter,
    toolRegistry: ToolRegistry,
    options?: OrchestratorOptions,
  ) {}

  async *run(goal: string, signal?: AbortSignal): AsyncGenerator<AgentEvent>
}
```

**Execution flow**:

```
run(goal)
│
├── Phase 1: Plan
│   └── Planner.plan(goal) → AgentStep[]
│       yield { type: 'plan', steps }
│
├── Phase 2: Execute steps (loop, maxIterations guard)
│   └── for each step:
│       ├── yield { type: 'step_start', step }
│       ├── executeStep(step, history, signal)
│       │   ├── If step.toolName + step.input → toolRegistry.execute() directly
│       │   └── Else → LLM call with available tools
│       │       ├── collect text chunks + tool_use chunks
│       │       └── execute any tool calls, attach results
│       ├── yield { type: 'step_complete', step }
│       └── (on error) yield { type: 'step_error' } — non-fatal, continues
│
└── Phase 3: Synthesize
    └── LLM call: summarize completed steps → goal outcome
        yield { type: 'text', content } for each chunk
        yield { type: 'done', task }
```

**Cancellation**: `signal?.aborted` is checked at the start of each phase and at the top of each step iteration. A cancelled run yields `{ type: 'error', error: 'Cancelled …' }` and sets `task.status = 'cancelled'`.

**Zero-step fallback**: If `Planner` returns an empty array (e.g. the goal is conversational), `run()` falls back to a direct LLM call and streams the response as `text` events.

**Server integration**: The orchestrator is implemented in `packages/core` only. Server-side route integration (exposing agent execution via a new API endpoint) is deferred to a future version.
