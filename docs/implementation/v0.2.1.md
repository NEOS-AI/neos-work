# v0.2.1 구현 문서 — 워크플로우 빌더 사용성 완성

## 구현 요약

v0.2.1은 v0.2.0에서 추가된 워크플로우 엔진과 에디터를 실제로 설정하고 실행할 수 있는 형태로 다듬었다. 핵심 변경은 선택 노드 설정 패널, 하네스/블록 선택 UI, 블록 파라미터 폼, 실행 전 검증, 실행 이력 runId 정합성, 엔진 회귀 테스트다.

## 1. 에디터 설정 패널

- `WorkflowEditor` 우측 패널을 `Config`, `Run Log`, `History` 탭으로 분리했다.
- 노드를 클릭하면 `Config` 탭으로 전환되고, 선택 노드의 label과 타입별 config를 바로 수정할 수 있다.
- Trigger, Web Search, Slack, Discord, Agent, Block, Gate, Output 노드의 최소 설정 UI를 제공한다.
- 실행 전 error-level validation이 있으면 실행을 막고 Config 탭에 검증 메시지를 표시한다.

## 2. 하네스/블록 선택 UI

- Agent 노드는 `HarnessSelector`로 도메인에 맞는 하네스를 선택할 수 있다.
- 선택한 하네스의 설명, 허용 도구, 실행 제약을 읽기 전용으로 요약한다.
- Block 노드는 `BlockSelector`로 등록된 블록을 선택할 수 있다.
- `BlockParamForm`은 `paramDefs`를 기반으로 string, number, boolean, select 입력을 렌더링한다.
- 블록 선택 시 기본 param 값을 채우고, 기존 입력값은 보존한다.

## 3. 그래프 검증

- `WorkflowValidation.ts`에 클라이언트 전용 검증 함수를 추가했다.
- 검증 항목은 label 누락, blockId 누락, 필수 block param 누락, harness 미선택 warning, web search query warning, Slack channel 누락, dangling edge, cycle, trigger/output 누락이다.
- 저장은 draft 상태를 허용하기 위해 막지 않고, 실행만 error-level validation에서 차단한다.

## 4. 실행 이력 정합성

- `executeWorkflow()`가 외부 `runId`를 받을 수 있게 `ExecutorOptions.runId`를 추가했다.
- 서버 workflow run route는 DB에 저장한 runId를 executor에 전달한다.
- SSE 이벤트의 runId와 DB run id가 같은 실행을 가리킨다.
- 클라이언트 abort 이후 저장 상태는 `cancelled`로 남기도록 정리했다.

## 5. 테스트

- `vitest`를 workflow-engine dev dependency로 추가하고 root `test` script를 추가했다.
- `graph.test.ts`에서 topological sort 정상 정렬과 cycle 감지를 검증한다.
- `executor.test.ts`에서 runId 주입과 blockId 없는 block 노드 실패를 검증한다.
- TDD RED 확인: runId 주입 구현 전 `uses the provided runId for SSE events`가 실패하는 것을 확인했다.

## 6. v0.2.0 대비 변경 사항

- 신규 빈 워크플로우 생성 시 Trigger와 Output 기본 노드 및 edge를 만든다.
- 워크플로우 빈 상태에서 템플릿 시작 CTA를 제공한다.
- 템플릿 카드에 필요한 설정 키 힌트를 표시한다.
- 실행 완료 후 History 탭이 최근 run 목록을 다시 불러온다.
- `apps/desktop/src/lib/engine.ts`의 workflow node type을 union 타입으로 구체화했다.

## 7. 남은 과제

- 일반 브라우저 dev 모드에서는 Tauri sidecar auth token이 없어 전체 API 기반 수동 플로우를 완전히 확인하기 어렵다.
- 프론트엔드 컴포넌트 테스트는 v0.2.2 범위로 남겼다.
- OR 게이트의 진짜 병렬 실행 semantics와 스케줄 트리거는 후속 버전 후보로 유지한다.
