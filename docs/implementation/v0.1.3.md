## 1. 브라우저 자동화 (`packages/browser-tool/`)

### 패키지 격리 이유
Playwright는 바이너리 의존성(Chromium)이 있어 core 패키지에 넣으면 서버 환경에서도 강제로 따라온다. `@neos-work/browser-tool`로 분리해서 에이전트 라우트가 필요할 때만 `import`하도록 했다.

### BrowserManager 설계
세션당 하나의 Chromium 인스턴스를 재사용한다. `connect()` → `getPage()` → `disconnect()` 라이프사이클로 관리하며, 서버 라우트의 `finally` 블록에서 `disconnect()`를 호출해 반드시 정리한다.

```
session.ts agent route
  ├── const browserManager = new BrowserManager()  ← try 밖 선언 필수 (finally 접근)
  ├── try { await browserManager.connect() ... }
  └── finally { await browserManager.disconnect() }
```

> **주의:** `browserManager`를 `try` 안에서 선언하면 `finally`에서 스코프 에러가 발생한다. TypeScript는 이를 컴파일 에러로 잡지 않으므로 런타임에만 드러난다.

### 6개 Tool

| 이름 | 입력 | 주요 옵션 |
|------|------|-----------|
| `browser_navigate` | `url` | `waitUntil: 'domcontentloaded'`, 30s 타임아웃 |
| `browser_click` | `selector` | 10s 타임아웃 |
| `browser_fill` | `selector`, `value` | 10s 타임아웃 |

| `browser_extract_links` | `selector?` | `{text, href}[]` 반환 |

## 2. Self-Healing Agent (`packages/core/src/agent/healing.ts`)

### 3단계 복구 흐름

```
step 실패
  ├── Attempt 1: RetryStrategy → 그대로 재실행
  ├── Attempt 2: ReflectionStrategy → LLM이 action/revisedStep 결정
  │     ├── action: retry   → revisedStep으로 재실행
  │     ├── action: skip    → 다음 step으로 진행 (non-fatal)
  │     └── action: abort   → 에이전트 전체 종료
  └── 모두 실패 → step_error (non-fatal, 다음 step 계속)
```

### ReflectionStrategy 구현 포인트
LLM 응답에서 JSON을 추출할 때 `response.match(/\{[\s\S]*\}/)` 패턴을 사용한다. LLM이 마크다운 코드펜스로 감싸거나 앞뒤에 설명을 붙이는 경우에도 JSON 블록만 파싱할 수 있다.

`action` 값이 예상 범위(`retry`, `skip`, `abort`) 밖이면 `skip`으로 폴백해서 에이전트가 멈추지 않도록 했다.

### SSE 이벤트: `step_healing`
```ts
// AgentEvent 타입에 추가
{ type: 'step_healing'; step: AgentStep; strategy: 'retry' | 'reflect' }
```
클라이언트는 이 이벤트로 "재시도 중…" / "반성 중…" UI를 표시한다.

## 3. Context Manager (`packages/core/src/llm/context-manager.ts`)

### 토큰 추정 방식
`chars / 4` 휴리스틱 사용. OpenAI tokenizer 기준 영어는 약 4자/토큰, 한국어는 약 2-3자/토큰이므로 실제보다 낮게 추정될 수 있다. 과소 추정이 과대 추정보다 낫다 (불필요한 압축 방지).

### 압축 전략
```
[전체 메시지]
  ├── 최근 20개 → 그대로 보존 (RECENT_WINDOW)
  └── 나머지 → LLM 요약 → system 메시지 1개로 대체
```

threshold 기본값 80,000 토큰 → Claude의 200k 컨텍스트 창 기준 40% 지점에서 압축 시작. 여유 있게 잡아서 압축이 너무 자주 일어나지 않도록 했다.

### 서버 통합 포인트
`session.ts` 채팅 라우트에서 LLM 호출 직전에 `contextManager.needsCompression()` 체크 후 `compress()`를 호출한다. 압축이 발생하면 SSE로 `{ type: 'context_compressed' }` 이벤트를 클라이언트에 전송한다.

## 4. 주요 설계 결정

### browser-tool 패키지를 pnpm workspace에 추가하는 절차
1. `packages/browser-tool/package.json`에 `"@neos-work/core": "workspace:*"` 의존성 추가
2. root `pnpm-workspace.yaml`에 `packages/browser-tool` 항목 추가
3. `apps/server/package.json`에 `"@neos-work/browser-tool": "workspace:*"` 추가
4. `pnpm install` → `pnpm build` (shared → core → browser-tool 순서)

### tsconfig.tsbuildinfo 주의
`tsc -p tsconfig.json` (빌드 모드)은 `.tsbuildinfo` 캐시를 생성한다. 다른 워크트리에서 빌드한 캐시가 남아 있으면 main 브랜치에서 `tsc --noEmit`이 "이미 최신"으로 오판할 수 있다. 빌드 문제 발생 시 `.tsbuildinfo` 삭제 후 재빌드.

### Healing이 non-fatal인 이유
에이전트 goal의 개별 step 실패가 전체 목표를 불가능하게 만들지 않는 경우가 많다 (예: 보조 정보 수집 실패). ReflectionStrategy가 `abort`를 결정할 때만 전체를 중단하고, `skip`/복구 실패는 다음 step으로 진행해 최대한 많은 결과를 만들어낸다.
